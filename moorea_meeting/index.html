Berkeley Ecoinformatics Engine

The goal of the development has been new ways of integrating ecological data that allows for:

  • Data discovery in context, e.g. vicinity of species, climatic envelopes, etc.
  • Enrichment of species records with environmental information derived from raster surfaces
  • Time and space are central concepts to relate heterogeneous data removed from context during a collection event
  • Search and discovery across different data types e.g. photos, drill cores, or even anthropological data

Current Data Assets

  • 5 million specimen records of all taxa
  • historic photos, indicating ecological change in CA
  • Wieslander Vegetation Type Mapping
  • Plot surveys (historic and recent)
  • Species checklists from field stations
  • Pollen records from drill cores (upcoming)
  • Historic soil surveys (upcoming)
  • Sensor data from field stations
  • Geophysical and climatic surfaces

Problems of data management in distributed ecological research

  • Standardization proofed to be very hard: GBIF, EML, metadata
  • Despite standards you have to do a lot of work to integrate data
  • Problem of data sharing, data aggregators, synching
  • Downloading copying
  • Measured against tech development, standards are outdated when agreed on

Problems of data delivery through web application

  • different life cycles of data and web pages
  • feature requested from different stakeholders make web sites often convoluted
  • a website not able to present all possible uses and audiences of a data sets
  • "locked-in data" (need to download for further analysis)

A solution:

  • strict separation between data services and frontend applications
  • RESTful APIs
    • One backend can support several general as well as special purpose applications
    • Interoperability through minimal standards based on http
    • Applications can draw on more than one single API
    • RESTful APIs are not a particular technology, they are an architectural principle (see Roy Fielding 2000) geared toward distributed computing
    • Most cloud services are using RESTful APIs to interact which each other: Google Maps, Facebook, ESRI server

Architecture

RESTful API

Application Programming Interface (API): A well defined and documented interface for applications to interact with each other.


RESTful APIs utilizing the http protocol. Users can usually communicate through well defined URL patterns (or by posting more complex data to the API using POST requests).


Example:

https://ecoengine.berkeley.edu/api/observations/?genus=puma

Advantages

  • Returns are machine-readable, ready to go for cloud computing
  • Data can be used in many contexts: web applications, software packages, desktop GIS
  • It is a open data sharing format that can be accessed in most coding languages
  • Web applications can be light-weight and focused on single purposes

Disadvantages

  • Latency and download speed provide limits to the concept of the web as a database
  • Requires still applications that aggregate returns, functionality that makes the API potentielly convoluted
  • Most suitable for slicing data and query by certain parameters or retrieve aggregated values, less useful if entire dataset is required
  • Requires some opportunism toward the technology

The social dimension

  • An API is technically not very hard to implement (there are very good packages out there: DjangoRestFramework)
  • Challenges are mostly of a social nature:
    • Definition of vocabulary, URL design, persistence, extensibility, and versioning
    • Develop API in parallel with teams that create consumers, test standards and communication
    • Data curation
    • Documentation is everything
    • Access control, and throttling policies

What is next?

  • Social API of sensors and events
  • Data Science within a http response cycle
  • Implement workflow steps to deliver modeling building blocks via API
